# Audit Jarvis Agent — 11 juillet 2026

Audit du code réel (pas de la doc) contre `docs/spec.md` et contre les extensions concurrentes citées en inspiration. Toutes les affirmations ci-dessous ont été vérifiées par lecture de code (fichiers/lignes cités).

---

## Résumé exécutif

Le projet est **beaucoup plus avancé que ce qu'on pourrait croire** en relisant seulement `spec.md` — les Phases 1 à 4 sont essentiellement complètes, et plusieurs items de la Phase 5 (Tree-sitter, RAG par embeddings, Cmd+K) sont déjà faits. Le câblage frontend↔backend est globalement propre (quasi tous les messages webview ont un handler côté extension et vice-versa).

Les vrais trous sont peu nombreux mais notables :
- Le concept de **"rôles" par modèle** (chat/edit/autocomplete/embed/...) est de la UI pure, jamais lue côté backend.
- **Aucune ouverture automatique** du fichier/diff après une édition par l'agent.
- **Pas de TODO visuel structuré** façon Claude Code (il y a un mode Plan et un log d'activité, mais pas de checklist persistante).
- **Pas d'autocomplete inline** (Tab) façon Continue.dev — le rôle existe dans la UI mais rien ne l'utilise.
- **Pas de fichier `JARVIS.md`/instructions projet** lu automatiquement.

### Écarts précis avec la spec

- **Agents spécialisés (§5.1)** : la spec décrit des agents avec des *outils dédiés* (QA-Agent → Linter/Tests/Coverage, Security-Agent → Scrubber/Audit...). En réalité, `services/agents.ts` définit 5 personas avec juste un `systemPrompt` différent — ils partagent **le même tool registry complet** que l'agent générique (`extension.ts:1547-1548`), sans restriction ni outil spécifique. C'est fonctionnel mais moins différencié que prévu.
- **Diff/Ouverture automatique** : la spec ne le demande pas explicitement, mais `view_diff` (§4.1) ne fait que retourner du texte — pas d'ouverture d'éditeur diff VS Code (voir §6 ci-dessous).
- **Web Search "sources configurables"** (§4.2) : l'outil `search_web` existe, mais rien n'indique un filtrage par source (Google/StackOverflow/MDN) — à vérifier si important pour toi.
- **Fichier de règles projet type CLAUDE.md** : la spec ne le mentionne pas explicitement mais c'est un existant chez les concurrents (Continue `.continuerules`, Claude Code `CLAUDE.md`) — actuellement les "rules" de Jarvis sont **100% pilotées par la config UI**, rien n'est lu depuis un fichier du repo (cf. §4).
- **Vectra** (cité en librairie section 10 de la spec) : n'est ni installé ni utilisé — le vrai store vectoriel est un tableau en mémoire dans `rag.ts`. Pas un problème (l'implémentation maison fonctionne), juste une note pour mettre à jour la doc de références.
- **Chart.js** (cité en §9.4) : installé mais **jamais utilisé** — l'AnalyticsPanel utilise des barres CSS pures, comme documenté dans CLAUDE.md. Dépendance à retirer du `package.json` si confirmé inutile ailleurs.

## 3. Idées inspirées de Claude Code / Mistral Vibe / Cline / Continue / Antigravity

Classées par ce qui manque vraiment (vs ce qui existe déjà et qu'il faudrait juste renforcer) :

**Manquant, à fort impact**
- **Autocomplete inline (Tab)** façon Continue.dev : `registerInlineCompletionItemProvider` n'existe nulle part dans le code. C'est la fonctionnalité phare de Continue et le champ "rôle autocomplete" existe déjà dans la UI (inutilisé, cf. §1) — c'est le morceau le plus simple à récupérer puisque la moitié de la plomberie de config est déjà là.
- **TODO list visuel structuré** façon Claude Code (`TodoWrite`) — voir détail en §5.
- **Hooks utilisateur** (pré/post tool-call, façon Claude Code `settings.json` hooks ou Cline) : permettrait par exemple d'auto-formatter après un edit, ou de bloquer certains outils par règle utilisateur sans repasser par HITL.
- **Sous-agents / délégation parallèle** : Claude Code peut lancer des agents en parallèle sur des sous-tâches indépendantes. Le `WorkflowRunner` de Jarvis est strictement séquentiel (`workflows.ts:121-169`) — pas de parallélisation possible actuellement.

**Manquant, impact modéré**
- **Règles par dossier** (Cline/Continue permettent des règles scopées à un sous-dossier, pas juste globales) — Jarvis n'a que des rules globales en config.
- **Suivi des coûts** (Cline affiche un coût $ estimé par requête basé sur le pricing du modèle cloud) — Jarvis compte les tokens mais pas de coût monétaire.
- **Checkpoints sur repo "fantôme"** (Cline utilise un second repo git caché pour ne jamais toucher à l'historique de l'utilisateur) — Jarvis utilise `git stash`/`git commit` sur le repo réel (cf. `checkpoint.ts`), ce qui est plus simple mais un peu plus intrusif ; à garder en tête si des utilisateurs se plaignent de bruit dans leur historique git.
- **Slash commands personnalisées en markdown** (Claude Code `.claude/commands/*.md`) — Jarvis a des workflows prédéfinis en dur et des custom tools JSON, mais pas de mécanisme "commande utilisateur = fichier markdown avec prompt".

**Déjà bien couvert** (ne pas dupliquer inutilement)
- MCP (serveurs externes configurables), custom tools JSON, sandboxing, scrubbing, checkpoints, HITL 3 modes, RAG par embeddings, Tree-sitter, Cmd+K, background process manager — c'est déjà au niveau ou au-dessus des extensions citées sur ces aspects précis.

## 4. Un fichier `JARVIS.md` façon `CLAUDE.md` — intéressant ?

**Oui, et c'est probablement le gain le plus simple de cet audit.** Actuellement, les "rules" projet sont **exclusivement** stockées dans la config utilisateur (`~/.jarvis/config.json` via les Settings, cf. `services/rules.ts`) — rien n'est lu depuis un fichier versionné dans le repo. Ça pose deux problèmes :
1. Les règles ne sont pas partagées entre membres d'une équipe (elles vivent dans la config locale de chacun).
2. Elles ne sont pas versionnées avec le code, donc pas d'historique/review sur leur évolution.

Un `JARVIS.md` à la racine du workspace, lu automatiquement et injecté dans `buildAgentSystemPrompt` (`orchestrator.ts:105-157`) aux côtés du `rulesText` existant, résoudrait ça sans rien casser : il suffirait de concaténer son contenu comme bloc supplémentaire (`PROJECT INSTRUCTIONS` avant ou après `USER RULES`). Coût d'implémentation faible (lecture fichier + cache + invalidation sur changement), bénéfice direct en cohérence d'équipe.

## 5. TODO visuel en mode Plan (comme Claude Code)

Ce qui existe déjà :
- Un vrai **mode Plan** (`runPlanMode`, `extension.ts:1045-1058`) : le modèle répond uniquement avec un plan markdown, sans écrire de fichiers, avec boutons "Proceed"/"Review" dans le chat (`ChatPanel.svelte:489-497`).
- Un **log d'activité éphémère** par message (`appendToProcessSteps`, `App.svelte:114-144`) qui accumule thinking/tool/step dans un `<details>` repliable au-dessus de la réponse — y compris les étapes de workflow (`Step {i}/{n}: {name}`).

Ce qui manque : ce log est **attaché à un message unique**, disparaît une fois le message rendu comme "fini", et n'est pas une checklist structurée avec statut par item (pending/in_progress/completed) affichée **au-dessus de la zone de saisie**, persistante pendant toute la durée d'une tâche multi-étapes — contrairement à `TodoWrite` de Claude Code.

Pour l'implémenter, il faudrait :
1. Un nouvel outil agent type `update_todo_list` (le modèle décide de la liste et des transitions de statut, comme Claude Code le fait).
2. Un store Svelte séparé (pas lié à un message de chat) affiché en position fixe au-dessus de l'input, avec cases à cocher.
3. Le vider/masquer à la fin de la tâche (ou le garder en historique repliable, à ta préférence).

C'est un ajout de complexité modérée mais qui améliorerait beaucoup la lisibilité des tâches longues — je te propose de le faire en tâche dédiée avec un plan si tu veux avancer dessus.

## 6. Auto-ouverture des fichiers modifiés après le diff

**Actuellement non implémenté.** Constat précis :
- `view_diff` (outil agent, `tool-registry.ts:333-338`) renvoie juste le texte de `git diff` dans le chat — aucun appel à `vscode.commands.executeCommand('vscode.diff', ...)` nulle part dans `src/backend` (recherche exhaustive, 0 résultat).
- `vscode.window.showTextDocument` n'est appelé que dans 3 endroits, aucun déclenché automatiquement après une édition de l'agent : export analytics, clic manuel sur un lien de fichier dans le chat (`ChatPanel.svelte:316` → `extension.ts:501`), et ouverture du fichier de config global.
- Ce qui existe déjà et va dans ton sens : `DiffDecorationProvider` (`core/diff-decorations.ts`) peint des décorations inline (vert pour ajouts, rouge "ghost text" pour suppressions) mais **uniquement dans les éditeurs déjà ouverts** — il ne force l'ouverture de rien.

**C'est faisable et peu risqué à ajouter.** Deux options, pas exclusives :
- **Option A (simple)** : après chaque `edit_existing_file`/`create_new_file` réussi, appeler `vscode.window.showTextDocument(uri)` pour amener le fichier au premier plan (l'utilisateur voit alors les décorations inline déjà en place).
- **Option B (plus proche de Cline/Continue)** : ouvrir un vrai éditeur diff via `vscode.diff` (avant/après) au lieu du fichier brut, particulièrement utile pour des gros changements.

Point d'attention : si l'agent modifie 10 fichiers dans une même tâche, ouvrir 10 onglets d'un coup peut être pénible — je recommande de le limiter (ex: seulement le dernier fichier modifié, ou un réglage utilisateur "toujours"/"jamais"/"si HITL mode strict").

## 7. "Run in background" — est-ce implémenté ?

**Oui, complètement.** `BackgroundProcessManager` (`src/backend/services/background-processes.ts`, ~121 lignes) :
- Lance en détaché (`cmd.exe /c` sous Windows), bufferise les 20 derniers Ko de stdout+stderr combinés, suit l'état running/exitCode.
- `stop()` tue tout l'arbre de processus (`taskkill /pid <pid> /T /F` sous Windows) — pas juste le shell parent.
- Exposé comme 3 outils agent : `run_in_background`, `check_background_process`, `stop_background_process` (`tool-registry.ts:280-331`).
- Le system prompt de l'agent l'instruit explicitement d'utiliser ça pour un serveur de dev/watcher (`orchestrator.ts:154`).

À distinguer de l'exécution "normale" (`run_terminal_command`, `mcp/tools/terminal.ts:27-111`) qui reste bloquante avec timeout (30s par défaut) — c'est le comportement attendu pour des commandes ponctuelles (`npm test`, `git status`), le mode background est réservé aux process longs. Rien à ajouter ici, c'est déjà au niveau des concurrents cités.

## Addendum — passe de vérification code + publication (11/07/2026, suite)

Les items 1 à 5 ci-dessus, et les items du §5.1 (agents avec restriction d'outils réelle) et §5.2 (`JARVIS.md`), ont depuis été implémentés — voir `docs/spec.md` §9.6 et `CHANGELOG.md`. Cette passe complémentaire a vérifié le code réel (build + lint + 322 tests, tous verts) et corrigé ce qui restait :

- **Props `onNewChat`/`onResumeChat` de `ChatPanel`** (finding #4 ci-dessus) : le CSS (`.empty-actions`/`.empty-action-btn`) et les props existaient déjà, mais **aucun bouton ne les utilisait dans le template** — le lint les signalait encore comme inutilisées malgré la note de `spec.md` §9.6 les disant "rendues fonctionnelles" (cette note était donc fausse). Deux boutons avaient d'abord été ajoutés à l'état vide du chat pour rendre la note exacte, puis retirés à la demande explicite de l'utilisateur (doublon non désiré avec les boutons déjà présents dans le header d'`App.svelte`). Décision finale : props `onNewChat`/`onResumeChat` et CSS orpheline **supprimés** de `ChatPanel.svelte`/`App.svelte` — la note de `spec.md` §9.6 a été corrigée en conséquence plutôt que rendue vraie.
- **5 erreurs ESLint réelles** (`no-empty` dans `orchestrator.ts`, `prefer-const` dans `extension.ts`/`ollama.ts`/`openai-compatible.ts`) — corrigées, non détectées par un simple `npm run build` (qui passait déjà).
- **Blocage de packaging** : `engines.vscode` (`^1.85.0`) était plus bas que `@types/vscode` (`^1.90.0`), ce qui fait échouer `vsce package` avec une erreur explicite. Relevé à `^1.90.0`.
- **Dépendances mortes/mal classées** : `uuid` (jamais importé nulle part dans `src/`) supprimé ; `marked`/`prismjs` déplacés de `dependencies` vers `devDependencies` (utilisés uniquement par le bundle webview via Vite, jamais par l'extension host — les avoir en `dependencies` gonflait inutilement le `.vsix` avec leur `node_modules`).
- **`vsce` déprécié** : le paquet `vsce` (v1.x) utilisé pour packager/publier est déprécié en faveur de `@vscode/vsce` et tire 4 vulnérabilités connues via ses dépendances transitives (`markdown-it`, `linkify-it`, `xml2js`). Migré vers `@vscode/vsce` — `npm audit` passe à 0 vulnérabilité.
- **`.vscodeignore` absent** : sans lui, `vsce package` embarque tout ce qui n'est pas dans `.gitignore` (sources TS, tests, docs). Créé, en prenant soin de ne **pas** exclure `node_modules/**` en bloc (l'extension n'étant pas bundlée par esbuild côté backend, les dépendances runtime doivent rester dans le `.vsix`).
- **Métadonnées `package.json` manquantes pour la publication** : `publisher` était le placeholder littéral `"your-publisher-name"`, et `repository`/`bugs`/`homepage`/`keywords`/`icon`/`galleryBanner` étaient absents. Complétés (voir liste de tâches de publication) ; `publisher` mis à `maxoubvd` par défaut (à vérifier/créer sur le Marketplace, ce n'est qu'une supposition basée sur le remote GitHub).
- **Icône Marketplace absente** : seul `media/jarvis.svg` existait (icône monochrome `currentColor` pour l'activity bar, pas utilisable telle quelle comme icône Marketplace qui doit être un PNG carré coloré). Un PNG 128×128 (`media/jarvis-icon.png`) a été généré à partir d'un SVG dérivé — **c'est un placeholder fonctionnel, pas un logo final** ; à remplacer par un vrai design si besoin.
- **`docs/media/` vide** : `README.md` référençait 6 images/vidéos (`logo_jarvis.png`, `overview_chat_interface.png`, `demo_cmd_k.mp4`, `rag_context_pruning.png`, `demo_tdd_loop.mp4`, `onboarding_welcome.png`, `settings_panel.png`) qui n'ont jamais existé dans le dépôt (seul un `.gitkeep`) — le rendu GitHub du README aurait affiché des images cassées. Références retirées, remplacées par une note explicite ; capturer ces médias reste une tâche humaine (voir liste de tâches).
- **`.env`/`.env.example` non utilisés par le code** : aucun `process.env.*` n'est lu dans `src/` — la config réelle des clés API passe uniquement par `~/.jarvis/config.json` (Settings UI). Ces fichiers sont un reliquat du scaffold initial ; à clarifier ou supprimer si confirmé inutile (non fait automatiquement, pas certain que rien d'externe n'en dépende).

Aucune régression : `npm run build`, `npm run lint` (0 erreur, 6 warnings préexistants sur `any`/import de test inutilisé) et `npm run test` (42 fichiers, 322 tests) passent après ces changements.
