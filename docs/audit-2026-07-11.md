# Jarvis Agent Audit — July 11, 2026

Audit of the actual code (not the docs) against `docs/spec.md` and against the competing extensions cited as inspiration. All claims below were verified by reading the code (files/lines cited).

---

## Executive summary

The project is **much further along than you'd think** from just re-reading `spec.md` — Phases 1 through 4 are essentially complete, and several Phase 5 items (Tree-sitter, embeddings-based RAG, Cmd+K) are already done. The frontend↔backend wiring is generally clean (almost every webview message has a handler on the extension side and vice versa).

The real gaps are few but notable:
- The concept of **per-model "roles"** (chat/edit/autocomplete/embed/...) is pure UI, never read on the backend.
- **No automatic opening** of the file/diff after an edit by the agent.
- **No structured visual TODO** Claude Code-style (there's a Plan mode and an activity log, but no persistent checklist).
- **No inline autocomplete** (Tab) Continue.dev-style — the role exists in the UI but nothing uses it.
- **No `JARVIS.md`/project instructions file** read automatically.

### Precise gaps vs the spec

- **Specialized agents (§5.1)**: the spec describes agents with *dedicated tools* (QA-Agent → Linter/Tests/Coverage, Security-Agent → Scrubber/Audit...). In reality, `services/agents.ts` defines 5 personas with just a different `systemPrompt` — they share **the exact same full tool registry** as the generic agent (`extension.ts:1547-1548`), with no restriction or specific tool. Functional, but less differentiated than planned.
- **Diff/Auto-open**: the spec doesn't explicitly ask for this, but `view_diff` (§4.1) just returns text — no VS Code diff editor opening (see §6 below).
- **Web Search "configurable sources"** (§4.2): the `search_web` tool exists, but there's no indication of filtering by source (Google/StackOverflow/MDN) — worth checking if this matters to you.
- **CLAUDE.md-style project rules file**: the spec doesn't explicitly mention it but it's an existing pattern among competitors (Continue's `.continuerules`, Claude Code's `CLAUDE.md`) — currently Jarvis's "rules" are **100% driven by the UI config**, nothing is read from a file in the repo (see §4).
- **Vectra** (cited as a library in spec section 10): neither installed nor used — the real vector store is an in-memory array in `rag.ts`. Not a problem (the homegrown implementation works), just a note to update the reference docs.
- **Chart.js** (cited in §9.4): installed but **never used** — AnalyticsPanel uses pure CSS bars, as documented in CLAUDE.md. Dependency to remove from `package.json` if confirmed unused elsewhere.

## 3. Ideas inspired by Claude Code / Mistral Vibe / Cline / Continue / Antigravity

Sorted by what's genuinely missing (vs. what already exists and would just need reinforcing):

**Missing, high impact**
- **Inline autocomplete (Tab)** Continue.dev-style: `registerInlineCompletionItemProvider` doesn't exist anywhere in the code. This is Continue's flagship feature and the "autocomplete role" field already exists in the UI (unused, see §1) — this is the easiest piece to pick up since half the config plumbing is already there.
- **Structured visual TODO list** Claude Code-style (`TodoWrite`) — see detail in §5.
- **User hooks** (pre/post tool-call, Claude Code `settings.json` hooks-style or Cline-style): would allow, for example, auto-formatting after an edit, or blocking certain tools via user rule without going through HITL.
- **Sub-agents / parallel delegation**: Claude Code can run agents in parallel on independent sub-tasks. Jarvis's `WorkflowRunner` is strictly sequential (`workflows.ts:121-169`) — no parallelization currently possible.

**Missing, moderate impact**
- **Folder-scoped rules** (Cline/Continue allow rules scoped to a subfolder, not just global ones) — Jarvis only has global rules in config.
- **Cost tracking** (Cline shows an estimated $ cost per request based on cloud model pricing) — Jarvis counts tokens but has no monetary cost tracking.
- **"Shadow" repo checkpoints** (Cline uses a second hidden git repo to never touch the user's history) — Jarvis uses `git stash`/`git commit` on the real repo (see `checkpoint.ts`), which is simpler but a bit more intrusive; keep in mind if users complain about noise in their git history.
- **Custom markdown slash commands** (Claude Code's `.claude/commands/*.md`) — Jarvis has hardcoded predefined workflows and JSON custom tools, but no "user command = markdown file with a prompt" mechanism.

**Already well covered** (no need to duplicate)
- MCP (configurable external servers), JSON custom tools, sandboxing, scrubbing, checkpoints, 3-mode HITL, embeddings-based RAG, Tree-sitter, Cmd+K, background process manager — already at or above the level of the cited extensions on these specific points.

## 4. A `JARVIS.md` file, CLAUDE.md-style — worth it?

**Yes, and it's probably the easiest win in this audit.** Currently, project "rules" are stored **exclusively** in user config (`~/.jarvis/config.json` via Settings, see `services/rules.ts`) — nothing is read from a file versioned in the repo. This causes two problems:
1. Rules aren't shared between team members (they live in each person's local config).
2. They aren't versioned with the code, so no history/review on how they evolve.

A `JARVIS.md` at the workspace root, read automatically and injected into `buildAgentSystemPrompt` (`orchestrator.ts:105-157`) alongside the existing `rulesText`, would fix this without breaking anything: just concatenate its content as an extra block (`PROJECT INSTRUCTIONS` before or after `USER RULES`). Low implementation cost (file read + cache + invalidation on change), direct benefit in team consistency.

## 5. Visual TODO in Plan mode (like Claude Code)

What already exists:
- A real **Plan mode** (`runPlanMode`, `extension.ts:1045-1058`): the model replies only with a markdown plan, without writing files, with "Proceed"/"Review" buttons in the chat (`ChatPanel.svelte:489-497`).
- An **ephemeral activity log** per message (`appendToProcessSteps`, `App.svelte:114-144`) that accumulates thinking/tool/step entries in a collapsible `<details>` above the response — including workflow steps (`Step {i}/{n}: {name}`).

What's missing: this log is **attached to a single message**, disappears once the message is rendered as "done", and is not a structured checklist with a per-item status (pending/in_progress/completed) displayed **above the input box**, persistent throughout a multi-step task — unlike Claude Code's `TodoWrite`.

To implement it, we'd need:
1. A new agent tool `update_todo_list` (the model decides the list and status transitions, the way Claude Code does).
2. A separate Svelte store (not tied to a chat message) displayed in a fixed position above the input, with checkboxes.
3. Clear/hide it at the end of the task (or keep it as collapsible history, your preference).

This is a moderate-complexity addition but would greatly improve readability for long tasks — happy to do it as a dedicated task with a plan if you want to move forward on it.

## 6. Auto-opening modified files after the diff

**Currently not implemented.** Precise findings:
- `view_diff` (agent tool, `tool-registry.ts:333-338`) just returns the `git diff` text in the chat — no call to `vscode.commands.executeCommand('vscode.diff', ...)` anywhere in `src/backend` (exhaustive search, 0 results).
- `vscode.window.showTextDocument` is only called in 3 places, none triggered automatically after an agent edit: analytics export, manual click on a file link in the chat (`ChatPanel.svelte:316` → `extension.ts:501`), and opening the global config file.
- What already exists and goes in your direction: `DiffDecorationProvider` (`core/diff-decorations.ts`) paints inline decorations (green for additions, red "ghost text" for deletions) but **only in already-open editors** — it doesn't force anything open.

**Feasible and low-risk to add.** Two options, not mutually exclusive:
- **Option A (simple)**: after each successful `edit_existing_file`/`create_new_file`, call `vscode.window.showTextDocument(uri)` to bring the file to the foreground (the user then sees the inline decorations already in place).
- **Option B (closer to Cline/Continue)**: open a real diff editor via `vscode.diff` (before/after) instead of the raw file, particularly useful for large changes.

Point of attention: if the agent modifies 10 files in a single task, opening 10 tabs at once could be annoying — I recommend limiting it (e.g. only the last modified file, or a user setting "always"/"never"/"if HITL strict mode").

## 7. "Run in background" — is it implemented?

**Yes, completely.** `BackgroundProcessManager` (`src/backend/services/background-processes.ts`, ~121 lines):
- Launches detached (`cmd.exe /c` on Windows), buffers the last 20 KB of combined stdout+stderr, tracks running/exitCode state.
- `stop()` kills the whole process tree (`taskkill /pid <pid> /T /F` on Windows) — not just the parent shell.
- Exposed as 3 agent tools: `run_in_background`, `check_background_process`, `stop_background_process` (`tool-registry.ts:280-331`).
- The agent's system prompt explicitly instructs it to use this for a dev server/watcher (`orchestrator.ts:154`).

To be distinguished from "normal" execution (`run_terminal_command`, `mcp/tools/terminal.ts:27-111`) which remains blocking with a timeout (30s by default) — this is the expected behavior for one-off commands (`npm test`, `git status`), background mode is reserved for long-running processes. Nothing to add here, already at the level of the cited competitors.

## Addendum — code verification + publication pass (2026-07-11, continued)

Items 1 through 5 above, and the items from §5.1 (agents with real tool restrictions) and §5.2 (`JARVIS.md`), have since been implemented — see `docs/spec.md` §9.6 and `CHANGELOG.md`. This follow-up pass verified the actual code (build + lint + 322 tests, all green) and fixed what remained:

- **`ChatPanel`'s `onNewChat`/`onResumeChat` props** (finding #4 above): the CSS (`.empty-actions`/`.empty-action-btn`) and the props already existed, but **no button used them in the template** — lint still flagged them as unused despite `spec.md` §9.6's note claiming they'd been "made functional" (that note was therefore wrong). Two buttons were first added to the chat's empty state to make the note accurate, then removed at the user's explicit request (unwanted duplication with the buttons already present in `App.svelte`'s header). Final decision: `onNewChat`/`onResumeChat` props and orphaned CSS **removed** from `ChatPanel.svelte`/`App.svelte` — `spec.md` §9.6's note was corrected accordingly rather than made true.
- **5 genuine ESLint errors** (`no-empty` in `orchestrator.ts`, `prefer-const` in `extension.ts`/`ollama.ts`/`openai-compatible.ts`) — fixed, not caught by a plain `npm run build` (which already passed).
- **Packaging blocker**: `engines.vscode` (`^1.85.0`) was lower than `@types/vscode` (`^1.90.0`), which makes `vsce package` fail with an explicit error. Bumped to `^1.90.0`.
- **Dead/misplaced dependencies**: `uuid` (never imported anywhere in `src/`) removed; `marked`/`prismjs` moved from `dependencies` to `devDependencies` (used only by the webview bundle via Vite, never by the extension host — having them in `dependencies` needlessly bloated the `.vsix` with their `node_modules`).
- **Deprecated `vsce`**: the `vsce` package (v1.x) used to package/publish is deprecated in favor of `@vscode/vsce` and pulls in 4 known vulnerabilities via its transitive dependencies (`markdown-it`, `linkify-it`, `xml2js`). Migrated to `@vscode/vsce` — `npm audit` now reports 0 vulnerabilities.
- **Missing `.vscodeignore`**: without it, `vsce package` bundles everything not in `.gitignore` (TS sources, tests, docs). Created, taking care **not** to blanket-exclude `node_modules/**` (since the extension isn't bundled by esbuild on the backend side, runtime dependencies must stay in the `.vsix`).
- **Missing `package.json` metadata for publishing**: `publisher` was the literal placeholder `"your-publisher-name"`, and `repository`/`bugs`/`homepage`/`keywords`/`icon`/`galleryBanner` were absent. Filled in (see the publication task list); `publisher` set to `maxoubvd` by default (to be verified/created on the Marketplace, this is just an assumption based on the GitHub remote).
- **Missing Marketplace icon**: only `media/jarvis.png` existed (a monochrome `currentColor` icon for the activity bar, not usable as-is as a Marketplace icon, which must be a colored square PNG). A 128×128 PNG (`media/jarvis-icon.png`) was generated from a derived SVG — **this is a functional placeholder, not a final logo**; replace with a real design if needed.
- **Empty `docs/media/`**: `README.md` referenced 6 images/videos (`logo_jarvis.png`, `overview_chat_interface.png`, `demo_cmd_k.mp4`, `rag_context_pruning.png`, `demo_tdd_loop.mp4`, `onboarding_welcome.png`, `settings_panel.png`) that never existed in the repo (only a `.gitkeep`) — GitHub's README rendering would have shown broken images. References removed, replaced with an explicit note; capturing this media remains a human task (see task list).
- **`.env`/`.env.example` unused by the code**: no `process.env.*` is read anywhere in `src/` — the actual API key config goes exclusively through `~/.jarvis/config.json` (Settings UI). These files are leftovers from the initial scaffold; to clarify or remove if confirmed unused (not done automatically, not certain nothing external depends on them).

No regressions: `npm run build`, `npm run lint` (0 errors, 6 pre-existing warnings on `any`/unused test import) and `npm run test` (42 files, 322 tests) pass after these changes.
